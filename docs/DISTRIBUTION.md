# Distribution — install channels and how they're published

getdev ships as **one static binary** for six targets. Everything below is generated and
published from a single tagged release by `cargo-dist` + GitHub Actions
(see [RELEASING.md](RELEASING.md)); no channel is hand-maintained except the Scoop manifest.

## Channel matrix

| Channel | User command | Published by | Priority |
|---|---|---|---|
| install script (macOS/Linux) | `curl -fsSL https://getdev.ai/install.sh \| sh` | cargo-dist `shell` installer, hosted on getdev.ai | v0.1 launch |
| install script (Windows) | `irm https://getdev.ai/install.ps1 \| iex` | cargo-dist `powershell` installer | v0.1 launch |
| **npm** | `npx getdev` / `npm i -g getdev` | cargo-dist `npm` installer → npmjs.com | v0.1 launch — **critical: vibe coders live in npm** |
| **Homebrew** | `brew install pzelenin/tap/getdev` | cargo-dist `homebrew` installer → tap repo | v0.1 launch |
| GitHub Releases | manual download | cargo-dist (canonical artifact source) | v0.1 launch |
| Scoop (Windows) | `scoop bucket add getdev https://github.com/pzelenin/scoop-bucket; scoop install getdev` | manifest in bucket repo, bumped by release workflow | v0.1 launch |
| crates.io | `cargo install getdev` | `cargo publish` step in release workflow | v0.1 launch |
| cargo-binstall | `cargo binstall getdev` | free — reads cargo-dist metadata from GitHub Releases | automatic |
| self-update | `getdev update` | built-in updater pulling from GitHub Releases (signature-checked) | v0.1 |
| GitHub Action | `uses: pzelenin/getdev-action@v1` | thin wrapper repo installing the binary + running `check` | v0.2 |
| winget / MSI | — | explicitly out of scope (scoop suffices for the audience) | — |

## Canonical source of truth

**GitHub Releases is the canonical artifact source.** Every other channel is a pointer to,
or repackaging of, those artifacts. Each release contains, per target:

- `getdev-<target>.tar.xz` / `.zip` (Windows) — the binary
- `getdev-<target>.tar.xz.sha256` — checksum
- cosign signature bundle + SLSA provenance attestation
- `getdev-<version>.sbom.spdx.json` — SBOM (Syft)
- `install.sh`, `install.ps1` — generated installers
- `dist-manifest.json` — cargo-dist metadata (consumed by cargo-binstall and the npm shim)

## Per-channel details

### install.sh / install.ps1 (getdev.ai)

Generated by cargo-dist for each release. getdev.ai (the landing/portfolio site, separate
repo: [`pzelenin/getdev`](https://github.com/pzelenin/getdev)) serves a copy of the latest
release's scripts and lists the CLI on its tools page; the site is only updated **after**
the release gate passes and checksums are verified — so `curl | sh` users can never receive
a half-published release. The scripts:
detect OS/arch → download from GitHub Releases → verify SHA-256 → install to
`~/.getdev/bin` (or `$GETDEV_INSTALL_DIR`) → add to PATH with the user's consent.
Version-pinned install: `curl -fsSL https://getdev.ai/install.sh | sh -s -- --version 0.1.2`.

### npm — package `getdev`

The npm wrapper is the highest-leverage channel: the target audience already has Node.
cargo-dist's npm installer generates a package that, on install, downloads the correct
platform binary from GitHub Releases and exposes the `getdev` bin — the esbuild pattern,
no Rust toolchain involved.

- Package name: **`getdev`** (reserve on npmjs.com before launch — see
  [Name reservations](#name-reservations)).
- Published by the release workflow with `NPM_TOKEN` (npm automation token, 2FA-exempt).
- `npx getdev check` must work in a clean container — this exact invocation is a release
  smoke test.
- The wrapper package contains no logic beyond download+verify; its version always equals
  the binary version.

### Homebrew — tap `pzelenin/homebrew-tap`

cargo-dist generates the formula (named `getdev`) and pushes it to the tap repo on each
release (`HOMEBREW_TAP_TOKEN`: a fine-grained PAT with write access to the tap repo only).

- Install: `brew install pzelenin/tap/getdev`
- The formula uses `bottle :unneeded`-style prebuilt binaries (cargo-dist downloads the
  release artifact; no compile on user machines).
- Post-v1.0 with adoption numbers, submit to **homebrew-core** so plain
  `brew install getdev` works; until then the tap is the supported path.

### Scoop — bucket `pzelenin/scoop-bucket`

cargo-dist does not manage Scoop; the bucket repo holds a JSON manifest with an
`autoupdate` stanza (URL template + hash extraction from the `.sha256` files), and the
release workflow bumps `version`/hashes via a small script step. Signed Windows binaries
matter here: unsigned fresh EXEs trip SmartScreen/AV heuristics (tracked risk in the plan).

### crates.io

`cargo install getdev` requires the *binary crate* to be published under the name `getdev`.
Therefore the crate in `crates/getdev-cli/` has `package.name = "getdev"` (directory name is
just workspace layout). Publish order matters — dependencies first:

```
getdev-grammars → getdev-core, getdev-registry, getdev-gitx → getdev
```

Handled by `cargo publish` steps in the release workflow (`CARGO_REGISTRY_TOKEN`); use
`cargo release`/`release-plz` if manual ordering gets error-prone. All workspace crates are
published (a `cargo install` build resolves them from crates.io, not git).

### cargo-binstall

Free once cargo-dist metadata is in the release: binstall reads `dist-manifest.json` and
downloads the prebuilt artifact instead of compiling. Verify once per release cycle that
`cargo binstall getdev` resolves.

### Self-update (`getdev update`)

The binary updates itself from GitHub Releases: check latest → download platform artifact →
**verify checksum + cosign signature** → atomic swap. Respects `--offline` (no-op with a
notice) and an update channel setting in `~/.getdev/config.toml`. This is one of only three
permitted network destinations (npm registry, PyPI, GitHub Releases) — keep it that way.

## Name reservations (do before anything is public)

- [x] npm: `getdev` — reserved 2026-07-09 (placeholder 0.0.1)
- [x] npm org `getdev` — created 2026-07-09; locks the `@getdev/*` scope against
      official-looking impostor packages (nothing published under it; owning it is the point)
- [x] crates.io: `getdev` — reserved 2026-07-09 (placeholder 0.0.0); `getdev-core`,
      `getdev-registry`, `getdev-gitx`, `getdev-grammars` publish naturally with v0.1
- [x] PyPI: `getdev` — reserved 2026-07-09 (permanent defensive stub, source in
      `packaging/reserve/pypi-getdev/`; we ship no Python package, but the audience will
      `pip install getdev` and an attacker shouldn't own that)
- [x] GitHub repos under `pzelenin`: `getdev` (getdev.ai landing/portfolio site — exists),
      `getdev-cli` (this CLI); still to create: `homebrew-tap`, `scoop-bucket`,
      `getdev-action` (v0.2)
- [x] Domain: getdev.ai (held) — plus wordmark/trademark search before launch

Naming convention: the GitHub *repo* is `getdev-cli` (the plain `getdev` repo is the
website), but the product name everywhere users type or see it — binary, Homebrew formula,
npm package, crates.io crate, release artifacts — is **`getdev`**. If a dedicated GitHub
org is created post-launch, repos move; package names don't.

## Support matrix

| OS | Arch | Target triple | Status |
|---|---|---|---|
| macOS 12+ | arm64 | `aarch64-apple-darwin` | release |
| macOS 12+ | x86_64 | `x86_64-apple-darwin` | release |
| Linux (glibc 2.31+) | x86_64 | `x86_64-unknown-linux-gnu` | release |
| Linux (glibc 2.31+) | arm64 | `aarch64-unknown-linux-gnu` | release |
| Linux (musl/Alpine) | x86_64 | `x86_64-unknown-linux-musl` | v0.2 candidate (CI images) |
| Windows 10+ | x86_64 | `x86_64-pc-windows-msvc` | release |
| Windows 11 | arm64 | `aarch64-pc-windows-msvc` | best-effort |

Every "release"-status target is exercised by the release smoke test (install → `getdev
doctor` → `getdev check` on a corpus project) before install.sh is repointed.
