# CI/CD â€” GitHub Actions setup

Two workflows: `ci.yml` (every push/PR â€” correctness) and `release.yml` (tags â€” artifacts).
Target: full CI matrix under **10 minutes** via sccache + dependency caching.

## `.github/workflows/ci.yml`

Jobs (all required for merge):

| Job | Runner(s) | What |
|---|---|---|
| `fmt` | ubuntu | `cargo fmt --check` |
| `clippy` | ubuntu | `cargo clippy --workspace --all-targets -- -D warnings` |
| `test` | ubuntu, macos, windows | `cargo test --workspace` + corpus integration tests |
| `docker-build-gate` | ubuntu **only** | `getdev ship --write --target docker` then `docker build` over each `testdata/fixtures/ship/*` preset (REQ-cmd-ship SC3) |
| `deny` | ubuntu | `cargo deny check` (licenses + RustSec advisories) |
| `coverage` | ubuntu | `cargo llvm-cov --workspace` â€” fail under 80% on library crates |
| `bench` | ubuntu (pinned runner type) | criterion against perf budgets; regression = red |
| `docs` | ubuntu | `cargo doc --no-deps` warnings-clean; link check on `docs/*.md` |

Reference skeleton:

```yaml
name: ci
on:
  push: { branches: [main] }
  pull_request:
permissions:
  contents: read
env:
  CARGO_TERM_COLOR: always
  RUSTC_WRAPPER: sccache
  SCCACHE_GHA_ENABLED: "true"
jobs:
  test:
    strategy:
      fail-fast: false
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable   # respects rust-toolchain.toml
      - uses: mozilla-actions/sccache-action@v0.0.9
      - uses: Swatinem/rust-cache@v2
      - run: cargo test --workspace
      - run: cargo test -p getdev-cli --test corpus
  # fmt / clippy / deny / coverage / bench jobs follow the same pattern
```

Notes:

- **Toolchain** comes from `rust-toolchain.toml` â€” never hardcode a version in the workflow.
- The **docker-build-gate** is the enforceable REQ-cmd-ship SC3 exit gate: it builds the
  `getdev` binary, runs `getdev ship --write --target docker` in each
  `testdata/fixtures/ship/<preset>/` (Node/Next.js, FastAPI, Flask, Django â€” each a minimal
  BUT complete app with a real lockfile/entrypoint, distinct from the non-buildable
  `testdata/corpus/seeded/*` AST fixtures), then `docker build`s the generated Dockerfile.
  It runs on **ubuntu-latest only** â€” GitHub-hosted macOS/Windows runners do not reliably
  support Linux container builds â€” and matrixes over the four presets so a single preset's
  failure is isolated.
- **aarch64** is not in the PR matrix (cost); it's covered by release-artifact smoke tests.
- The **bench** job runs on one pinned runner class; treat >10% regressions as failures but
  expect noise â€” compare against a stored baseline (`criterion --save-baseline`), don't
  assert absolute wall-clock on shared runners.
- Corpus tests must not hit the network: `real` tests run against a recorded registry
  cache / local mock, and CI sets `GETDEV_OFFLINE=1` for everything except the dedicated
  registry-client tests (which use mocks anyway). A CI run must be hermetic.
- Windows job is a first-class citizen (CRLF, path separators, ANSI) â€” never mark it
  optional "temporarily".

## `.github/workflows/release.yml`

**Generated by `dist generate --mode ci`** from `dist-workspace.toml` â€” do not hand-edit the
generated sections; regenerate on cargo-dist upgrades and review the diff. Trigger: tags
matching `v*`. Full behavior documented in [RELEASING.md](RELEASING.md).

Required permissions on the release workflow (the generated jobs set `attestations`/`id-token`
per-job for the attestation step; the custom self-update-signing job needs `contents: write`):

```yaml
permissions:
  contents: write        # create the release, upload artifacts (incl. SHA256SUMS + .sig + SBOM)
  id-token: write        # SLSA provenance attestation (keyless OIDC) â€” set on the build job
  attestations: write    # SLSA provenance â€” set on the build job
```

**npm publish + Homebrew tap push are cargo-dist-native** (driven by `installers`/`publish-jobs`
in `dist-workspace.toml`) â€” not hand-rolled. The genuinely custom (non-generated) steps, appended
AFTER the generated sections in a clearly-delimited `# --- getdev custom` block so a future
`dist generate` never clobbers them:

- aggregate the per-artifact `.sha256` files into a single `SHA256SUMS` manifest;
- **keyed** cosign sign-blob over `SHA256SUMS` â†’ `SHA256SUMS.sig` (the artifact `getdev update`
  verifies with its embedded p256 key, D-01) â€” pins **cosign 2.x** via `sigstore/cosign-installer`
  (3.x emits the incompatible bundle format), consumes `COSIGN_PRIVATE_KEY`/`COSIGN_PASSWORD`;
- Syft SBOM via `anchore/sbom-action` (a Go binary â€” **never** a `syft-rs` crate, which does not
  exist);
- a binary-size gate that fails the release if any target binary is â‰Ą 25 MB;
- Scoop manifest bump â†’ `getdev-ai/scoop-bucket` (`SCOOP_BUCKET_TOKEN`);
- ordered `cargo publish` (grammars â†’ core/registry/gitx â†’ getdev, `CARGO_REGISTRY_TOKEN`).

Secrets these need are listed in
[RELEASING.md](RELEASING.md#required-repository-secrets); grant them to the workflow via a
`release` **environment** with required-reviewer protection, so a compromised PR can't
exfiltrate the publish tokens or the `COSIGN_PRIVATE_KEY` signing key.

## `.github/dependabot.yml`

```yaml
version: 2
updates:
  - package-ecosystem: cargo
    directory: /
    schedule: { interval: weekly }
    groups:
      minor-and-patch:
        update-types: ["minor", "patch"]
  - package-ecosystem: github-actions
    directory: /
    schedule: { interval: weekly }
```

`cargo-deny` in CI is the security backstop between Dependabot runs.

## Repository settings (one-time)

- **Branch protection on `main`**: require PRs; required checks = fmt, clippy, testĂ—3,
  docker-build-gateĂ—4, deny, coverage; require linear history (merge queue or squash); no
  force-push.
- **DCO enforcement**: enable the DCO check app (or the `dco` GitHub Action) as a required check.
- **Actions hardening**: pin third-party actions by SHA; default `permissions: contents: read`
  at workflow level; disable `pull_request_target` usage entirely (getdev scans untrusted
  repos â€” its own CI must not be an injection vector).
- **Private vulnerability reporting**: enabled (pairs with [../SECURITY.md](../SECURITY.md)).
- **Tag protection**: `v*` tags restricted to maintainers.

## Local parity

Everything CI runs is runnable locally with stock cargo commands (see
[BUILDING.md](BUILDING.md#lint--hygiene-ci-denies-warnings)) â€” if it's green locally on your
OS, the only usual surprises are Windows paths and snapshot drift.
