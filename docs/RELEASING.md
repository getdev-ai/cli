# Releasing getdev

How a release goes from tag to every install channel. Audience: maintainers.
Related: [DISTRIBUTION.md](DISTRIBUTION.md) (channels), [CI.md](CI.md) (workflows),
[BUILDING.md](BUILDING.md) (local builds).

## Versioning

- **SemVer from v0.1.0.** Pre-1.0: breaking CLI/config/schema changes bump the minor;
  fixes bump the patch. The findings `schema_version` is versioned independently and
  declared stable at v1.0.
- All workspace crates share one version (`workspace.package.version`); the tag is `vX.Y.Z`.
- **Conventional commits are mandatory** — the changelog is generated by `git-cliff` from
  commit history; a sloppy commit message becomes a sloppy changelog entry.

## The release gate (blocking — every release, no exceptions)

1. **CI matrix green**: fmt, clippy `-D warnings`, tests on ubuntu/macos/windows,
   `cargo-deny`, coverage ≥ 80% on library crates.
2. **Benchmarks within budget** (criterion): `check` warm < 3 s, cold < 15 s,
   `audit`/`review` < 2 s, `snap` < 1 s on the reference corpus; binary < 25 MB.
3. **Corpus recall/FP targets**: 100% seeded-defect recall for `real`/`env`; per-rule
   recall/FP thresholds met; no rule above 5% FP on the sentinel set at its shipped severity.
4. **`docker build` succeeds** for every `ship` preset output.
5. **Manual "first five minutes" smoke** on all 3 OSes from actual release artifacts:
   install → `init` → `check` → `env --write` → `snap`/`back`.
6. Artifacts signed, checksummed, SBOM attached; **only then** repoint getdev.ai installers.

## Release procedure

```bash
# 0. on main, clean tree, CI green
git switch main && git pull

# 1. bump version (all crates, via workspace.package.version)
#    update Cargo.toml + `cargo update -w` so Cargo.lock follows
#
#    RELEASE-TIME TOGGLE: all five crates ship with `publish = false` as a
#    pre-release guard. For a real v0.1 crates.io release, flip every crate's
#    `publish = false` -> `publish = true` (getdev-grammars, getdev-core,
#    getdev-registry, getdev-gitx, and getdev — crates/getdev-cli) in THIS
#    release commit. The release workflow's ordered `cargo publish` step assumes
#    the guard is off; leaving it on makes that step fail safely (the intended
#    behavior for an accidental pre-release tag). Never flip it on `main` outside
#    a tagged release.

# 2. changelog
git cliff --tag v0.1.0 -o CHANGELOG.md
git add -A && git commit -s -m "chore(release): v0.1.0"

# 3. sanity-check what cargo-dist will do
cargo dist plan

# 4. tag and push — this triggers .github/workflows/release.yml
git tag v0.1.0
git push origin main v0.1.0
```

From the tag, the release workflow (generated by `cargo dist init`, config in
`dist-workspace.toml`) does the rest:

```
tag v0.1.0
 ├─ build matrix: 5 targets (macOS arm64/x64, Linux x64/arm64 gnu, Windows x64)
 │                 [+ aarch64-pc-windows-msvc — best-effort, commented in dist-workspace.toml]
 ├─ per artifact: SHA-256 checksum (.sha256)
 ├─ binary-size gate: fail the release if any target binary ≥ 25 MB
 ├─ attest: SLSA provenance via GitHub artifact attestations (keyless OIDC, no secrets)
 ├─ create GitHub Release (draft) + upload artifacts + git-cliff notes
 │  — getdev custom (not cargo-dist-managed) —
 ├─ aggregate per-artifact .sha256 → single SHA256SUMS manifest
 ├─ sign (KEYED): cosign sign-blob --key SHA256SUMS → SHA256SUMS.sig  (COSIGN_PRIVATE_KEY/PASSWORD)
 │                — the exact artifact `getdev update` verifies with its embedded p256 key (D-01)
 ├─ SBOM: Syft (anchore/sbom-action) → getdev-<v>.sbom.spdx.json  (never a `syft-rs` crate — it doesn't exist)
 ├─ upload SHA256SUMS + SHA256SUMS.sig + SBOM to the release
 │  — cargo-dist-native publish jobs —
 ├─ publish npm package `getdev`                (NPM_TOKEN)
 ├─ push Homebrew formula → getdev-ai/homebrew-tap    (HOMEBREW_TAP_TOKEN)
 │  — getdev custom (not cargo-dist-managed) —
 ├─ bump Scoop manifest → getdev-ai/scoop-bucket      (SCOOP_BUCKET_TOKEN)
 └─ cargo publish (ordered): getdev-grammars → core/registry/gitx → getdev   (CARGO_REGISTRY_TOKEN)
```

### Two signing paths — keyed self-update vs. keyless manual-verify

getdev uses cosign in **two distinct ways**, and they must not be confused:

- **Keyed self-update signature (automated, D-01).** The release workflow keyed-signs the
  aggregated `SHA256SUMS` manifest with `cosign sign-blob --key env://COSIGN_PRIVATE_KEY`
  (a **cosign 2.x** legacy detached signature — base64 ASN.1-DER ECDSA-P256 over
  `sha256(SHA256SUMS)`). `getdev update` verifies `SHA256SUMS.sig` **in-process** against the
  embedded public key (`EMBEDDED_COSIGN_PUBKEY`, wired to the real release key in 08-08) using
  pure-Rust `p256` — **no `cosign` binary, no network, no Rekor** (see
  `crates/getdev-cli/src/update/signature.rs`). The CI signing step MUST pin cosign to a 2.x
  release: cosign 3.x defaults to the new Sigstore *bundle* format (and a Rekor round-trip),
  which `verify_detached` cannot read. This is the self-update trust anchor.
- **Keyless manual-verify signature/attestation (humans & CI, v0.2 upgrade path).** GitHub
  artifact attestations (`github-attestations = true`) provide SLSA provenance bound to the
  workflow's OIDC identity — verifiable with `gh attestation verify` / `cosign verify-blob`
  (see [Verifying a release](#verifying-a-release)). This is a **separate, secretless** layer
  for humans; it is NOT what self-update consumes. Migrating self-update to keyless cosign
  (Fulcio/Rekor) is the **v0.2** direction (D-01), not wired in v0.1.

### After the workflow completes

1. Run the smoke checklist (below) against the **draft** release artifacts.
2. Publish the GitHub Release (undraft).
3. Repoint getdev.ai `install.sh`/`install.ps1` at the new version (checksum-verified
   deploy — a PR to the website repo) and update the CLI listing there
   if the release changes user-facing surface.
4. Verify each channel:
   - `curl -fsSL https://getdev.ai/install.sh | sh` in a clean container
   - `npx getdev@latest version`
   - `brew update && brew install getdev-ai/tap/getdev`
   - `scoop update getdev` (Windows VM)
   - `cargo binstall getdev` resolves the prebuilt artifact
   - `getdev update` from the previous version self-updates and signature-verifies
5. Announce (release notes; launch-window posts per plan §11).

## dist-workspace.toml (reference configuration)

```toml
[workspace]
members = ["cargo:."]

[dist]
cargo-dist-version = "0.32.0"
targets = [
  "aarch64-apple-darwin", "x86_64-apple-darwin",
  "x86_64-unknown-linux-gnu", "aarch64-unknown-linux-gnu",
  "x86_64-pc-windows-msvc",
  # aarch64-pc-windows-msvc — best-effort (support matrix); kept commented so a
  # build failure on it never blocks a release. A maintainer may enable it.
]
installers = ["shell", "powershell", "npm", "homebrew"]
tap = "getdev-ai/homebrew-tap"
npm-package = "getdev"
publish-jobs = ["homebrew", "npm"]   # cargo-dist-native publish; Scoop + crates.io are custom
ci = "github"
install-path = "~/.getdev/bin"
checksum = "sha256"
# SLSA provenance (keyless) — distinct from the keyed self-update signature (D-01, custom step)
github-attestations = true
```

(Exact keys track the pinned cargo-dist version; regenerate the workflow with
`dist generate --mode ci` on upgrades and review the diff — the keyed-cosign / SBOM / size-gate
/ Scoop / ordered-`cargo publish` steps live in a clearly-delimited custom block AFTER the
generated sections, so regeneration never clobbers them.)

## Verifying a release

Users and CI can verify any artifact:

```bash
# checksum
shasum -a 256 -c getdev-aarch64-apple-darwin.tar.xz.sha256

# cosign keyless signature (identity = the release workflow, issuer = GitHub OIDC)
cosign verify-blob \
  --certificate-identity-regexp 'github.com/getdev-ai/cli/.github/workflows/release.yml' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --bundle getdev-aarch64-apple-darwin.tar.xz.sigstore.json \
  getdev-aarch64-apple-darwin.tar.xz

# SLSA provenance via GitHub attestations
gh attestation verify getdev-aarch64-apple-darwin.tar.xz --repo getdev-ai/cli
```

## Hotfix releases

Branch from the release tag (`git switch -c hotfix/v0.1.1 v0.1.0`), cherry-pick the fix,
run the **full** gate (no shortcuts — the gate exists precisely for the releases you're
tempted to rush), tag `v0.1.1` from the hotfix branch, then merge back to main.

## Yanking a bad release

1. Repoint getdev.ai installers at the previous good version (first — this is the widest pipe).
2. Mark the GitHub Release as pre-release + edit notes with the warning; do not delete
   artifacts (checksums are referenced elsewhere).
3. `npm deprecate getdev@X.Y.Z "reason"` · revert the tap formula commit · revert the scoop
   manifest bump · `cargo yank --version X.Y.Z getdev` (and affected lib crates).
4. `getdev update` reads latest non-prerelease — marking pre-release stops self-update rollout.
5. Post-mortem → new patch release through the normal gate.

## Required repository secrets

| Secret | Used for | Scope notes |
|---|---|---|
| `COSIGN_PRIVATE_KEY` | keyed self-update signature (`cosign sign-blob --key` over `SHA256SUMS`) | the keyed private half of the cosign key pair whose public half is embedded in the binary (`EMBEDDED_COSIGN_PUBKEY`); generated + provisioned in 08-08 |
| `COSIGN_PASSWORD` | passphrase protecting `COSIGN_PRIVATE_KEY` | provisioned alongside the key in 08-08 |
| `NPM_TOKEN` | npm publish | automation token, `getdev` package only |
| `HOMEBREW_TAP_TOKEN` | tap formula push | fine-grained PAT, write to `homebrew-tap` only |
| `SCOOP_BUCKET_TOKEN` | scoop manifest bump | fine-grained PAT, write to `scoop-bucket` only |
| `CARGO_REGISTRY_TOKEN` | crates.io publish | scoped to the five getdev crates |

**SLSA provenance** (GitHub attestations) needs **no secrets** — it uses the workflow's OIDC
identity (`id-token: write` / `attestations: write`). The **keyed self-update signature**, by
contrast, *does* require the long-lived `COSIGN_PRIVATE_KEY`/`COSIGN_PASSWORD` (D-01) — these
are the only release secrets that sign, so they are gated behind the `release` environment with
required-reviewer protection (a compromised PR must never reach them). Guard the private key as
the highest-value secret in the repo: it is the self-update trust anchor.
